🎯
Objectif : Plateforme SaaS cloud-native de gestion du fret et de l'entrepôt — concurrent direct de CargoWise et Magaya — avec une UX supérieure, une architecture multi-tenant robuste, et un modèle de données mode-agnostic (ocean + air + road dès le jour 1).
⭐
3 Différenciateurs UX clés :
1. Exception Inbox Zero — navigation clavier J/K/E/S, "Event → Rule → Task" jamais "Event → Widget"
2. Kanban drag-and-drop shipments — glisser = transition state machine + audit + toast
3. Omni-search Cmd+K — global sur shipments, bookings, contacts, invoices

Les 5 Phases

Monorepo Turborepo, Docker Compose, 40+ tables Prisma, Row-Level Security, Auth JWT/SSO/2FA. Base solide avant toute feature métier.

2.2 Tariffs & Pricing
Semaines 3–4
  • CRUD Tariffs par carrier / lane / mode / période de validité

  • Méthodes : PER_CONTAINER | PER_KG | PERCENT_OF_BASE | FLAT

  • Snapshot FX rate au booking (jamais live-floating — quotes historiques reproductibles)
  • Margin rules par client / lane / mode / carrier
  • Calculateur prix itemisé (jamais total seul)
  • NEWVolumétrique métrique air : (L × W × H cm) ÷ divisor (défaut 6000, configurable tenant+carrier)
  • NEWVolumétrique impérial air : (L × W × H inches) ÷ 366 (ou 194 selon carrier)
  • NEWbillable_weight = MAX(actual_weight, volumetric_weight) — convention air standard
  • Unit switcher persistant kg/cm ↔ lb/in (user preference, sans rechargement)
  • Import Excel/CSV tariffs + alertes expiration (T-30j, T-7j)

2.5 Exception Queue & Inbox Zero ⭐ CŒUR MVP
Semaines 8–9
⚡
"Event → Rule → Task" — jamais "Event → Dashboard Widget". Queue mode-agnostic et source-agnostic : carrier EDI, visibility aggregators, IoT telemetry → même enveloppe d'exception.
  • Rules engine JSON par tenant (versioned, sans redéploiement)
  • ⭐ UXInbox Zero UI : navigation clavier J/K (haut/bas) · E (résoudre) · S (snooze)
  • Groupage par urgence SLA : CRITICAL → HIGH → MEDIUM → LOW
  • Indicateur visuel SLA restant (barre rouge si < 25%)
  • Snooze configurable (1h · 4h · 24h · custom)
  • Réassigner exception à un collègue
  • Auto-escalade Slack/Teams webhook (SLA à 50%)
  • Auto-escalade email (SLA à 75%)
  • Commentaires & historique par exception

2.6 WMS — Warehouse Management
Semaines 10–11
  • Gestion entrepôts, zones, allées, niveaux (grid_x / grid_y / grid_z)
  • ASN (Advance Shipment Notices) depuis booking ou EDI 214
  • ⭐ UXPWA mobile scan réception (BarcodeDetector API — caméra, sans app native)
  • Fallback BLE HID scanner (input caché capturant keystrokes — hardware-agnostic)
  • Mode offline PWA avec sync différée · Vibration haptic confirmation
  • Génération barcodes GS1-128 : {tenant_prefix}{shipment_id_short}{piece_sequence}{check_digit}
  • Unicité barcode par tenant + fenêtre glissante 90 jours
  • Algorithme slotting greedy (nearest free bin au outbound dock)
  • Discrepances (short/over/damaged) → tâche auto exception queue (pas rapport enfoui)
  • CQRS : scan_events append-only → projector async → warehouse_locations.occupied_cbm
  • Optimistic concurrency (version column) — pas de long-held locks
  • Gestion ULDs — cycle de vie indépendant des shipments (ULD peut porter plusieurs shipments)
  • NEWCross-dock : inbound scan → match manifest → staging lane assignée
  • NEWdwell_minutes = scanned_out_at − received_at calculé automatiquement
  • NEWDépassement target_dwell_minutes → exception CROSS_DOCK_DWELL_EXCEEDED (escalade rapide)
  • Interface plan d'entrepôt visuel (grille interactive)

3.1 IoT Telemetry Ingestion NOUVEAU
Semaines 15–16
⚡
Pipeline séparé du bus business-events. IoT = haute fréquence (secondes), petits payloads. 99% du volume ne touche pas OLTP PostgreSQL. Seul le breach d'un threshold devient un business event.
Device MQTT → AWS IoT Core / Azure IoT Hub → Kinesis Data Streams / Azure Event Hub → Stream processor (Kinesis Analytics / Apache Flink) ├─ raw readings → TimescaleDB hypertable (time-series store) └─ threshold breach seulement → event telemetry.reefer.temp_excursion → Kafka / AWS EventBridge (bus business-events) → exception_queue type TEMP_EXCURSION → Inbox Zero UI (même que tous les autres types)
  • Setup AWS IoT Core ou Azure IoT Hub (MQTT ingestion)
  • Kinesis Data Streams / Azure Event Hub
  • Stream processor Flink / Kinesis Analytics
  • TimescaleDB hypertable pour telemetry_readings
  • Table telemetry_devices (device_type, serial, shipment_id, tenant_id)
  • Threshold rules configurables par tenant + shipment (plage temp reefer, zone GPS)
  • Breach → event telemetry.reefer.temp_excursion → exception_queue
  • Graphique température/humidité sur page shipment (time-series chart)
  • Alertes GPS sortie de zone prédéfinie → exception

Principes d'Architecture
Décisions clés et invariants du système
🔐
Multi-tenancy : RLS PostgreSQL + middleware tenant résolution sur 100% des requêtes. tenant_id dans toutes les cache keys sans exception.
🆔
IDs : ULID sur tous les shipments (pas auto-increment — évite énumération tenant cross-region).
💰
Multi-devise : Toujours stocker amount_original + fx_rate_to_functional + amount_functional. Snapshot FX au moment du booking, jamais live-floating.
🔄
Idempotency : Tout consumer vérifie processed_events avant side-effect. Webhooks : HMAC-SHA256 + event_id unique.
📡
IoT séparé : Pipeline Kinesis/Flink indépendant du bus business-events Kafka. Seuls les breaches de threshold traversent vers PostgreSQL.
⚖️
Compliance as data : Les règles compliance sont des lignes dans compliance_rules — ajouter un pays = INSERT, pas un déploiement.
